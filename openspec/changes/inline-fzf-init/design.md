## Context

`dot_zshrc.tmpl` currently initializes fzf via `[ -f ~/.fzf.zsh ] && source ~/.fzf.zsh`. The `~/.fzf.zsh` file is a 7-line shim generated by fzf's install script that adds fzf to PATH and loads keybindings/completions. Every other CLI tool (starship, zoxide, atuin) initializes inline. This inconsistency makes the repo not fully self-describing — the fzf init is a black box to anyone reading just the repo.

Current `~/.fzf.zsh` contents:

```bash
if [[ ! "$PATH" == */usr/local/opt/fzf/bin* ]]; then
  PATH="${PATH:+${PATH}:}/usr/local/opt/fzf/bin"
fi
source <(fzf --zsh)
```

## Goals / Non-Goals

**Goals:**

- Make fzf initialization inline in `dot_zshrc.tmpl`, consistent with other tools
- Handle arch differences (ARM `/opt/homebrew/opt/fzf/bin` vs Intel `/usr/local/opt/fzf/bin`) using chezmoi templates
- Keep the repo fully self-describing with no references to unmanaged external files

**Non-Goals:**

- Removing or cleaning up existing `~/.fzf.zsh` files on machines — they become harmless dead files
- Supporting non-macOS fzf installations (Linux typically installs fzf to a system PATH location and doesn't need a PATH shim)
- Changing fzf configuration (FZF_DEFAULT_OPTS, FZF_DEFAULT_COMMAND, etc.) — those are separate and already inline

## Decisions

### Use `source <(fzf --zsh)` instead of static keybinding/completion files

**Rationale**: This is the modern fzf API (0.48+). It dynamically generates the correct keybindings and completions for the installed version. No static files to manage or go stale.

**Alternative considered**: Copying the full keybinding/completion scripts inline. Rejected because it would create a maintenance burden and desync with fzf updates.

### Use chezmoi OS/arch conditionals for PATH, matching existing pattern

**Rationale**: `dot_zshrc.tmpl` already uses `{{ if eq .chezmoi.os "darwin" }}` / `{{ if eq .chezmoi.arch "arm64" }}` for zsh plugin paths (lines 79-90). The fzf PATH addition should follow the exact same pattern for consistency.

**Alternative considered**: Using a single `eval "$(brew --prefix)/opt/fzf/bin"` to auto-detect. Rejected because `brew --prefix` adds ~200ms shell startup latency and the pattern of static template conditionals is already established.

### Place fzf PATH and init where the current `source` line is

**Rationale**: The fzf block (lines 52-70) is already well-organized. The PATH addition and `source <(fzf --zsh)` replace lines 52-53 directly. The existing FZF_DEFAULT_COMMAND/FZF_DEFAULT_OPTS blocks remain untouched.

## Risks / Trade-offs

- **[fzf < 0.48]** → `source <(fzf --zsh)` requires fzf >= 0.48. Mitigation: Homebrew satisfies this requirement, and the install script (`run_once_install-packages.sh.tmpl`) already uses brew. Verify with `fzf --version`.
- **[Dead file]** → Existing `~/.fzf.zsh` files on machines remain but are no longer sourced. Mitigation: Harmless — the file sits unused. No cleanup needed.
- **[Linux]** → The PATH shim is macOS-only (brew paths). On Linux, fzf installed via package manager is already in PATH. The `source <(fzf --zsh)` line works on both. Mitigation: The template conditional only wraps the PATH block, not the `source` line.
